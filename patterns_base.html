База знаний — интеграционные паттерны

Паттерны интеграции: Канал и Сервис

Паттерн — проверенный способ решить типовую проблему. У каждого паттерна есть уровень (где в архитектуре применяется) и категория (какую задачу решает).

Канал
Как физически доставить данные от системы А к системе Б?
Сервис
Как выстроить логику, коммуникацию и устойчивость?
Данные
Как обеспечить согласованность и единую версию правды?

Уровень Канал

Инфраструктура связности. Прежде чем думать о логике или надёжности — нужно выбрать по какому каналу пойдут данные. Закрывает проблемы: лавина N×N связей, нет контрактов, нет управляемой ИБ.

Категория: Управляемая связность каналов
📋
Message Broker (Kafka / MOM)
Асинхронный посредник между отправителем и получателем
БАЗА сложность: средняя ▼
Аналогия: доска объявлений в офисе. Менеджер повесил "Создан заказ №123" и ушёл. Склад, финансы и логистика читают доску каждый в своём темпе. Менеджер не ждёт и не знает кто читает.
Без брокера
Отправитель ждёт ответа: упал получатель — встал отправитель
Новый потребитель = изменения у отправителя
Пик нагрузки ломает получателя
Одна упавшая система тянет за собой всех
С брокером
Положил в топик и пошёл дальше — не ждёт
Новый потребитель просто подписывается на топик
Kafka буферизует пик, потребители читают в своём темпе
Упавший потребитель — сообщения ждут в топике
Пример: интернет-магазин создаёт заказ и публикует событие order.created в топик. Склад, финансы и уведомления подписаны на топик и реагируют независимо. Добавился новый сервис аналитики — просто подписался, ничего не меняли.
Для аналитика: при описании интеграции фиксируй название топика и формат события. Топики лучше называть по бизнес-событию: order.created, customer.updated — не по системе-источнику.
Инструменты:Apache KafkaRabbitMQ
Говорим в эфир — слышит тот, кому надо
📄
Schema Registry — контракты сообщений
Хранилище версионированных схем для топиков Kafka
БАЗА сложность: низкая ▼
Аналогия: обязательный шаблон служебных записок. Все заявки по единой форме из реестра. Изменил форму — согласуй новую версию. Никто не получит неожиданный формат. Это как Swagger, но для Kafka: Swagger описывает REST API, Schema Registry описывает сообщения в топиках.
Без Schema Registry
Разработчик переименовал поле — потребители падают молча
Контракт живёт в документации и устаревает незаметно
Несовместимые изменения доходят до продакшна
С Schema Registry
Схема зарегистрирована и версионирована рядом с Kafka
Несовместимое изменение не пройдёт валидацию при отправке
Любой видит что именно летит в каждом топике
Пример: сервис заказов публикует событие со схемой v1: поля order_id, amount, status. Через месяц хотят переименовать amount в total_price. Без Registry — потребители упадут. С Registry — изменение заблокируется как несовместимое, нужно выпустить схему v2 и дать потребителям время обновиться.
Для аналитика: при описании интеграции через Kafka в ПАРе ссылка на схему в Registry сильнее чем ссылка на страницу в Confluence — схема живая и версионированная.
Инструменты:Confluent Schema RegistryAsyncAPIOpenAPI
Swagger — для REST. Schema Registry — для Kafka
🚪
API Gateway — единая входная точка
Все внешние запросы идут через один шлюз
БАЗА сложность: средняя ▼
Аналогия: ресепшн в бизнес-центре. Все посетители снаружи приходят на ресепшн. Он проверяет кто вы, пускает или нет, записывает визит, направляет в нужный кабинет. Сотрудники внутри об этом не думают.
Без Gateway
Каждый сервис сам проверяет токены, права, лимиты
Новый внешний канал = изменения в каждом сервисе
Логи вызовов разбросаны, нет единой картины
С Gateway
Аутентификация и rate-limiting в одном месте
Новый внешний канал — правило в Gateway, сервисы не трогаем
Единые метрики всех входящих вызовов
Пример: при выводе монолита (Strangler Fig) Gateway становится точкой переключения трафика. Все системы обращаются к Gateway, он решает куда направить: в старый монолит или в новый сервис. Переключение через feature flags без изменений у потребителей.
Для аналитика: Gateway окупается когда много внешних потребителей в одну точку, или при миграции монолита. При небольшом числе внешних интеграций один-в-один — может быть избыточен.
Инструменты:KongGravitee
Один вход — много сервисов внутри
🔄
Mediation / Transformation — преобразование в канале
Посредник трансформирует формат, протокол или содержание данных
БАЗА сложность: средняя ▼
Аналогия: переводчик на переговорах. Стороны говорят на разных языках, переводчик в середине. Никто не учит чужой язык — этим занимается переводчик. Стороны не меняются.
Без Mediation
Источник знает форматы всех получателей и трансформирует сам
Получатели разбирают чужой формат — лишняя зависимость
Добавился новый получатель — меняем источник
С Mediation
Источник отдаёт в своём формате, посредник трансформирует
Каждый получатель получает в своём формате
Системы не знают друг о друге
Три вида трансформации:

Формат — XML на входе, JSON на выходе. Поле называется customer_id в источнике, а получатель ждёт КодКлиента.

Протокол — источник отдаёт по REST, получатель умеет только SFTP. Mediation принимает REST и кладёт файл.

Содержание — источник отдаёт полную запись, получателю нужны только определённые поля. Или нужно добавить поле которого нет в источнике — например подставить код склада из справочника по региону из заказа.
Для аналитика: маппинг полей который ты описываешь в требованиях — это и есть задача Mediation. Фиксируй: что на входе, что на выходе, правила трансформации, откуда берутся значения которых нет в источнике.
Инструменты:Apache CamelApache NiFiGateway plugins
Переводчик в канале — системы говорят на своих языках
📦
MFT — управляемый файловый обмен
Защищённый шлюз для файлов с внешними партнёрами
B2B сложность: низкая ▼
Аналогия: защищённый КПП для внешних курьеров. Партнёр не заходит в офис сам — оставляет посылку на КПП. Там проверяют, записывают в журнал, выдают подтверждение и только потом пускают внутрь.
Голый FTP
Файл от партнёра попадает сразу внутрь без проверок
Нет подтверждения доставки партнёру
Нет аудита: кто что положил и когда
Данные идут открытым текстом
С MFT
Файл проходит антивирусную проверку до попадания внутрь
Автоматическое подтверждение доставки
Полный аудит каждой передачи в обе стороны
SFTP или AS2 — шифрование и электронная подпись
Про протоколы:

SFTP — FTP с шифрованием. Минимальный уровень безопасности, данные не летят открытым текстом.

AS2 — с электронной подписью. Партнёр не может сказать "не получали" — есть криптографическое подтверждение факта передачи. Актуален когда нужно юридическое подтверждение или данные чувствительные.
Для аналитика: при описании интеграции с внешним партнёром через файлы в ПАРе раздел 7.5 (ИБ) спросит: как файлы проверяются на входе, есть ли аудит, как аутентифицируется партнёр, нужна ли электронная подпись.
Инструменты:GoAnywhereOpenText
Файл снаружи — сначала через КПП, потом внутрь
Базовый словарь: режимы взаимодействия
🔀
Четыре режима канала: Sync, Async, Псевдо-sync, Псевдо-async
Режим выбирается по потребности клиента, а не по возможностям получателя
БАЗА ▼
Sync: отправитель ждёт ответа здесь и сейчас. Как телефонный звонок — пока не ответили, ждёшь. REST-запрос.

Async: отправитель не ждёт. Как сообщение в мессенджере — отправил и пошёл дальше. Kafka.

Псевдо-sync: снаружи выглядит как sync — клиент ждёт ответа. Внутри async — сервис положил задачу в очередь и ждёт пока другой сервис обработает. Клиент не знает что внутри очередь.

Пример псевдо-sync: пользователь нажал "Оформить заказ" и ждёт подтверждения. Сервис заказов положил задачу в Kafka, сервис обработки забрал и ответил. Пользователю кажется обычный REST.

Псевдо-async: снаружи выглядит как async — клиент не ждёт, получит ответ потом. Внутри sync — обработка идёт сразу. Клиент получает результат через callback или вебхук.

Пример псевдо-async: вебхук от внешнего сервиса. Ты не ждёшь — сервис сам пришлёт событие когда оно случится. Внутри у него синхронная обработка.
Для аналитика: когда описываешь интеграцию — фиксируй режим взаимодействия. Это влияет на выбор паттерна, требования к надёжности и UX.
Режим — по потребности клиента, не по удобству получателя

Уровень Сервис

Логика, коммуникация и устойчивость. После того как выбран канал — нужно выстроить как сервисы взаимодействуют, как разбить монолит и что делать когда что-то идёт не так. Закрывает проблемы: хрупкость, монолит, нет наблюдаемости.

Категория: Декомпозиция и API
🌿
Strangler Fig — постепенный вывод монолита
Функции монолита заменяются новыми сервисами по одной, без остановки
B2B B2C сложность: высокая ▼
Аналогия: фикус-душитель в джунглях. Растёт рядом с деревом, постепенно оплетает и в итоге занимает его место. Старое дерево умирает само — новое уже стоит. Никакой революции, только постепенное замещение.
Переписать монолит целиком
Годы разработки
Бизнес не получает новых функций всё это время
Огромный риск: новая система может не заработать
Откат невозможен
Strangler Fig
Берёшь одну функцию, делаешь новый сервис
Переключаешь трафик постепенно через feature flags
Монолит продолжает работать
Откат в любой момент — вернул трафик в монолит
Как работает технически: все запросы идут через API Gateway. Gateway знает куда направить: в монолит или в новый сервис. Сначала 5% трафика на новый сервис — проверили, всё ок — 100%. Потребители не меняются, они всегда ходят в Gateway.
Для аналитика: при описании интеграций с монолитом фиксируй: является ли эта функция кандидатом на вынос. Если да — какой критерий готовности нового сервиса, какой порядок переключения, что является откатом.
Инструменты:API GatewayFeature flags
Не сносим — заменяем по куску
📱
BFF — Backend for Frontend
Отдельный backend под каждый тип пользовательского интерфейса
B2C сложность: средняя ▼
Аналогия: переводчик в посольстве. Один говорит с китайскими гостями по-китайски, другой с французскими по-французски. Внутри страны один и тот же закон. Каждый клиент получает информацию в удобном для него виде.
Один общий API
Мобайл получает лишние данные — тратит трафик пользователя
API обрастает параметрами под каждый клиент
Новый тип клиента = изменения в общем API
С BFF
Каждый клиент получает ровно то что нужно
Внутренние сервисы не знают кто их спрашивает
Новый тип клиента — новый BFF, остальное не трогаем
Пример: страница товара в интернет-магазине.

Веб-BFF собирает: название, все фото, описание, характеристики, отзывы, похожие товары, остаток, историю цены.

Мобайл-BFF собирает: название, одно фото, цену, кнопку купить. Всё.

Партнёр-BFF собирает: артикул, цену, остаток, вес, габариты — в техническом формате партнёра.

Внутренние сервисы товаров, склада и отзывов одни и те же. BFF — это адаптер между клиентом и внутренним миром.
Для аналитика: BFF про пользовательские интерфейсы — сайт, мобайл, личный кабинет. Для интеграций между системами не применяется. Актуален когда у одного API принципиально разные потребители с разными потребностями.
Инструменты:Node.jsGraphQLgRPC-Web
Свой backend — под язык каждого клиента
🧩
API Composition — сборщик ответов
Один сервис собирает данные из нескольких систем и отдаёт единый ответ
БАЗА сложность: средняя ▼
Аналогия: менеджер с готовой сводкой. Сам сходил в CRM, ERP и склад — тебе показал одну карточку. Ты не знаешь откуда данные, тебе не надо делать три звонка.
Без Composition
Клиент делает три запроса в три системы
Клиент сам склеивает ответ
Медленно, сложно, хрупко
С Composition
Один запрос — один ответ
Composition ходит в системы параллельно
Клиент не знает откуда данные
Пример: карточка клиента в CRM-интерфейсе. Менеджер открывает — нужны данные из трёх систем: профиль из CRM, история заказов из ERP, бонусный счёт из программы лояльности. Composition принимает один запрос, параллельно идёт в три системы, склеивает ответ и отдаёт одну карточку.
Для аналитика: когда видишь что для одного экрана нужны данные из нескольких систем — это кандидат на API Composition. Фиксируй: что нужно собрать, из каких источников, какая система является мастером для каждого куска данных.
Инструменты:GraphQL FederationApollo Router
Один ответ — собран из многих источников
Категория: Коммуникация
📡
Event-Driven — событийная архитектура
Сервисы реагируют на события, а не вызывают друг друга напрямую
БАЗА сложность: средняя ▼
Аналогия: корпоративная рассылка. HR публикует "вышел новый сотрудник". IT настраивает доступы, СБ выдаёт пропуск, бухгалтерия заводит в систему — каждый реагирует сам. HR не знает кто что делает и не ждёт подтверждения от каждого.
Без Event-Driven
Сервис A вызывает B, C и D напрямую
A знает о существовании B, C и D
Упал D — A тоже встал
С Event-Driven
A публикует событие и не знает кто его слушает
B, C и D реагируют независимо
Упал D — A и другие не пострадали
Пример: сервис заказов публикует событие order.created. На него реагируют: сервис склада (резервирует товар), сервис уведомлений (шлёт письмо клиенту), сервис аналитики (обновляет статистику). Сервис заказов не знает об их существовании.
Для аналитика: при описании бизнес-процессов думай о ключевых событиях: что происходит? Заказ создан, клиент зарегистрирован, оплата прошла. Это и есть кандидаты на события в Event-Driven архитектуре.
Инструменты:KafkaAsyncAPI
Говорим в эфир — слышит тот, кому надо
📢
Pub/Sub — публикация и подписка
Один производитель, много независимых подписчиков
БАЗА сложность: низкая ▼
Аналогия: Telegram-канал. Автор пишет один пост — его читают тысячи подписчиков независимо друг от друга. Автор не знает кто подписан. Каждый подписчик читает в своём темпе.
Без Pub/Sub
Производитель знает всех потребителей и шлёт каждому
Новый потребитель = изменения у производителя
С Pub/Sub
Производитель публикует в топик, не знает кто читает
Новый потребитель просто подписывается
Разница Event-Driven и Pub/Sub: Event-Driven — это принцип архитектуры (реагировать на события). Pub/Sub — это механизм доставки (один → многим через топик). Pub/Sub реализует Event-Driven. Kafka реализует Pub/Sub.
Инструменты:Kafka topicsNATS
Один → многим, без знания "кто слушает"
🔔
Async Request/Reply — асинхронный запрос-ответ
Клиент отправляет запрос и не ждёт — ответ придёт позже через callback или вебхук
B2C сложность: средняя ▼
Аналогия: заказ в МФЦ. Подал документы, получил номер и ушёл. Когда готово — пришла смс с тем же номером. Не нужно сидеть и ждать у окошка.
Sync для медленных операций
Клиент ждёт пока долгая операция завершится
Таймауты, зависшие соединения
Пик нагрузки ломает сервис
Async Request/Reply
Клиент получил номер запроса и ушёл
Ответ придёт через webhook или callback
Пики гасятся очередью
Пример — вебхук от внешнего сервиса: внешний сервис собирает пользовательские события. Вместо того чтобы вы периодически забирали файл с FTP, сервис сам присылает вебхук когда случилось событие. Вы получаете, кладёте в Kafka, отвечаете "принял". Обработка идёт асинхронно. Задержка — секунды вместо часов.
Для аналитика: когда видишь интеграцию через FTP с внешним сервисом который поддерживает вебхуки — это кандидат на замену. Вопрос: насколько критична скорость реакции на событие.
Инструменты:Kafka + correlation IDWebhook
Запрос ушёл, ответ заберём позже — по номеру
⛓
Saga — распределённая транзакция
Сквозной бизнес-процесс через несколько сервисов с возможностью отката
B2B сложность: высокая ▼
Аналогия: бронирование поездки. Шаг 1 — забронировал гостиницу. Шаг 2 — купил билет. Шаг 3 — оплата не прошла. Запускаем откат: отменяем билет, снимаем бронь. Каждый шаг знает как откатиться.
Без Saga
Шаг 1 прошёл, шаг 2 упал — данные рассогласованы
Откат вручную или никак
Непонятно в каком состоянии процесс
С Saga
Каждый шаг имеет компенсирующее действие
Откат запускается автоматически при сбое
Состояние процесса всегда известно
Для аналитика: Saga актуальна когда один бизнес-процесс затрагивает несколько систем и важна согласованность. Например: создать заказ + списать бонусы + зарезервировать товар + уведомить склад — всё должно произойти или ничего.
Инструменты:CamundaTemporalConductor
Шагнул не туда — есть кнопка "назад"
Категория: Надёжность

Circuit Breaker — предохранитель
Отключает вызовы к сбойному сервису, защищая от каскадного отказа
БАЗА сложность: низкая ▼
Аналогия: электрический автомат в щитке. При коротком замыкании сам отключается — спасает всю проводку. Остальные комнаты не обесточиваются из-за одной.
Без Circuit Breaker
Запрос к упавшему сервису висит до таймаута (30+ секунд)
100 запросов = 100 потоков висят и ждут
Ресурсы заканчиваются — падают живые сервисы
С Circuit Breaker
При пороге ошибок — мгновенный отказ без ожидания
Потоки свободны, остальные сервисы живут
Через время пробует снова — закрывается если сервис ожил
Три состояния: Закрыт (всё хорошо, запросы идут) → Открыт (ошибок много, сразу возвращает fallback) → Полуоткрыт (пробный запрос — если прошёл, закрывается обратно).

Fallback — заготовленный ответ когда предохранитель открыт. Сервис рекомендаций упал — показываем популярные товары. Сервис остатков недоступен — показываем "уточните наличие".

Применимость: только для REST. В Kafka не нужен — отправитель не ждёт ответа.
Для аналитика: фиксируй в требованиях что возвращать когда сервис недоступен (fallback). Это бизнесовое решение которое разработчик не примет сам.
Инструменты:Resilience4jIstio
Замкнуло — отключили, остальные живы

Retry с backoff — автоматический повтор
При сбое повторяет запрос с нарастающей паузой
БАЗА сложность: низкая ▼
Аналогия: звонок другу у которого занято. Не долбим без остановки — перезваниваем через минуту, потом через три, потом через пять. Большинство кратковременных сбоев решается само.
Без Retry
Кратковременная сетевая ошибка = сбой для пользователя
Заявка в техподдержку на проблему которая уже прошла
С Retry
200мс — 400мс — 800мс — большинство ошибок исчезают
Человек не вовлекается в кратковременные сбои
Важно — идемпотентность: Retry безопасен только если повторить запрос дважды = тот же результат что и один раз.

Получить данные — идемпотентно, повторять можно.
Создать заказ — не идемпотентно, повтор создаст два заказа.

Это нужно фиксировать в требованиях: можно ли применять Retry к этому вызову.
Для аналитика: при описании каждого REST-вызова фиксируй является ли операция идемпотентной. От этого зависит можно ли применять Retry.
Инструменты:Resilience4jSpring Retry
Не получилось? Попробуй ещё, но не сразу

Bulkhead — переборки
Изолирует потребителей через разделённые пулы ресурсов
B2B B2C сложность: средняя ▼
Аналогия: водонепроницаемые переборки на корабле. Пробоина в одном отсеке — корабль не тонет, вода не растекается.
Без Bulkhead
Медленный партнёр занял все 100 потоков сервиса
Быстрые внутренние системы встали в очередь
Один медленный потребитель роняет всех
С Bulkhead
Партнёру выделен свой пул — 20 потоков
Внутренним системам свой пул — 80 потоков
Партнёр тормозит в своём пуле, остальные не замечают
Когда актуален: когда у одного сервиса есть потребители с сильно разным профилем нагрузки. Медленный внешний партнёр и быстрые внутренние системы — классический кейс.
Инструменты:Thread/connection poolsIstio
Пробоина в отсеке — судно не тонет
Категория: Наблюдаемость
🔍
Distributed Tracing — сквозная трассировка
Каждому запросу присваивается ID, который проходит через все сервисы
БАЗА сложность: низкая ▼
Аналогия: трек-номер посылки. Видишь в каком городе посылка сейчас и где задержалась. Не нужно звонить на каждый склад по отдельности.
Пример: запрос прошёл через 5 сервисов и где-то завис. Без трассировки — нужно идти в логи каждого сервиса отдельно. С трассировкой — один поиск по trace ID показывает всю цепочку: где был каждый сервис, сколько времени потратил, на каком шаге задержка.
Для аналитика: при описании интеграции фиксируй требование — trace ID должен передаваться между сервисами. Без этого диагностика инцидентов занимает дни.
Инструменты:OpenTelemetryTempoJaeger
У запроса свой трек-номер
📝
Centralized Logging — централизованные логи
Все сервисы пишут логи не в файл на сервере, а в единое хранилище. При инциденте один поиск по trace ID — видна вся история по всем сервисам.
БАЗА

Health Check + Registry — проверка живости
Каждый сервис сообщает платформе "я жив" через /health эндпоинт. Балансировщик направляет трафик только на живые инстансы. При описании интеграции — фиксируй требование: сервис должен иметь /health эндпоинт.
БАЗА
🔒
Rate Limiting — лимиты запросов
Ограничение числа запросов от одного потребителя в минуту. Один шумный клиент или зациклившаяся система не роняет сервис для всех остальных.
B2C
🏍
Sidecar — вспомогательный контейнер
Рядом с каждым сервисом запускается помощник, который берёт на себя инфраструктурную рутину: шифрование, Retry, метрики. Разработчики пишут бизнес-логику, платформенная команда — инфраструктуру.
